# 文献调研与汇报撰写规范

## 适用范围
所有文献调研、技术汇报、方法比较等需要"写给用户看"的输出。包括飞书文档、讨论回复、分析报告。

## 核心原则：可复现级别的信息量

调研报告的读者（用户）需要读完后能**直接决策和复现**，而非再去读原文。"概要级"调研是不合格的。

## 文献调研标准

### 每篇重点文献必须包含

1. **完整数据流**：从输入到输出的完整维度链
   - 好：`InterPro (41311d) → EmbeddingBag → 1024d → 2层GCN 1024d → concat 2048d → FC → M classes`
   - 差：`用GCN处理PPI网络，然后分类`

2. **架构每个组件的参数**：层数、维度、激活函数、归一化、dropout、注意力头数
   - 好：`GraphSAGE + LSTM 聚合器，2层 (256→128→64)，BatchNorm，ReLU，Dropout=0.5`
   - 差：`用了GraphSAGE`

3. **损失函数公式及权重**：不只说"用了focal loss"，要给出 alpha、gamma、各损失的权重比
   - 好：`L = 0.4×L_gen + 0.6×L_con + 1.0×L_cls(focal, alpha=0.25, gamma=2)`
   - 差：`联合训练三个损失`

4. **训练超参数**：optimizer、lr、batch_size、epochs、early stopping
5. **消融实验具体数字**：每个变体的指标 + 与完整模型的 delta 百分比
6. **代码仓库链接**：GitHub URL、使用的框架（PyG/DGL/自定义）
7. **数据集统计**：节点数、边数、类别数、不平衡比

### 与当前项目方案的对照

每篇相关文献必须有**逐组件对比表**：

| 组件 | 本项目 (current) | 对比论文 | 分析 |
|------|------------------|---------|------|
| GNN | SAGEConv 5层 | GCN 2层 | 对方故意浅层避免过平滑 |
| ... | ... | ... | ... |

不要泛泛地说"对我们项目有启发"，要具体到**哪个组件可以怎么改**。

## 概念解释标准

用户问"这是什么"时，回答必须包含：
1. **一句话定义**
2. **直觉类比**（如：`生成式重建 = "别忘了你是什么蛋白"，类比考试前不只复习考点还要全面理解教材`）
3. **伪代码或公式**（如果涉及算法/训练流程）
4. **具体数字例子**（如：`x_ori[蛋白A] = [0.3, -0.1, 0.8, ...]，重建后 = [0.28, -0.05, 0.75, ...]`）

缩写首次出现必须展开（如 `SSL = Self-Supervised Learning（自监督学习）`）。

## 训练流程描述标准

涉及训练过程的方法，必须用**伪代码**展示完整的训练循环，标注：
- 每个步骤在哪个阶段执行（是每个batch？每个epoch？预处理？）
- 哪些模块共享参数、哪些独立
- 损失的计算顺序和反向传播范围

```
# 模板
for epoch in range(N):
    for batch in DataLoader:
        # 第0步：数据准备（说明采样策略）
        # 第1步：前向传播（说明哪些分支、是否共享参数）
        # 第2步：计算各损失（给出公式和权重）
        # 第3步：反向传播
```

## 数据库/工具说明标准

当提到某个数据库或工具时，回答"它自带这个特征吗？"这类问题：
- 说明数据来源（是数据库自带的？还是需要额外计算的？）
- 给出具体的文件名/字段名/API（如：`从 STRING 的 protein.links.detailed.v12.0.txt 文件获取`）
- 说明每个字段的含义和取值范围

## 引用铁律（零容忍，不可绕过）

**核心原则：先搜索、再读原文、再写结论。不允许凭记忆或训练数据写引用。**

### 文献纳入的三步验证流程（必须完整执行）

1. **搜索确认 DOI/PMID**：用 WebSearch 找到文献，确认 DOI 真实存在
2. **读原文核实内容**：用 WebFetch 读 PMC 全文或 abstract，找到**逐字引用的原文句子**，确认该句子确实支持我们的具体结论
3. **留痕记录**：在飞书文档的引用处，写明：
   - 完整引用格式（作者、年份、期刊、DOI）
   - 原文支持句（逐字引用，加引号）
   - 该句子支持了我们结论的哪一个具体点

**不允许行为**：
- 仅搜到标题/摘要就写入正文，不读内容
- 全文不可获取时，强行从摘要推断全文支持某结论
- 记录引用但不留原文支持句（用户无法复查）

**全文不可获取时的处理**：
- 若 PMC/PubMed 摘要已包含支持句 → 可引用，但必须注明"引自摘要"
- 若摘要不包含支持句 → 标记为 `[TODO: 全文待核实]`，不写入正文
- 绝不因"逻辑上应该支持"就直接纳入

### 必须先搜索文献的结论类型

| 结论类型 | 典型示例 | 执行要求 |
|---------|---------|---------|
| "与…机制/功能一致/吻合" | "与鞭毛互作网络高度吻合" | 每一个"吻合"都需要对应文献 |
| "一一对应" | "与各类别成员蛋白实际生物功能一一对应" | 每一对对应关系都需要独立引用 |
| "大多有文献支撑" | "上述模式大多有文献支撑" | 必须逐一列出支撑每个"大多"成员的文献 |
| 段落末尾拔高结论 | "表明模型学到了功能解析度特征" | 依赖的每个前述例子都必须已经有引用 |
| 具体蛋白/通路/机器的功能描述 | "MotA/B 是运动马达蛋白" | 需要引用描述该组件功能的结构或综述文章 |

### 执行流程（不可跳步）

```
1. 草拟结论句  →  2. 用 WebSearch 搜索验证  →  3. 确认内容匹配（不幻觉）
→  4. 按规范格式插入引用  →  5. 再写入文档
```

### 禁止行为

- **禁止**凭记忆写引用（AI 训练数据中的 PMID/DOI 可能是错的）
- **禁止**先写结论再"找文献凑数"（引用必须真正支持该具体结论）
- **禁止**用"大家都知道"/"生物学常识"代替引用
- **禁止**把文献放在段首统一引用后，用"如前所述"含糊带过每个子结论

### 引用格式规范

```
Author et al., Year, Journal (IF, if notable). DOI: xxx
# 示例：
Rajagopala SV et al., 2007. Mol Syst Biol, 3:128. DOI: 10.1038/msb4100166
```

### 验证检查点（写完后自检）

- [ ] 每个"与…一致/对应/吻合"后面跟着括号引用
- [ ] 没有连续三句生物学描述却无一引用
- [ ] "拔高结论"所依赖的具体例子都已单独引用
- [ ] 引用的 DOI/PMID 经过搜索验证（非凭记忆）

## 汇报文档组织

### 信息密度
- 宁可一篇文献写 500 字详解，不要 5 篇文献各写 100 字概要
- 用户说"感兴趣"的文献 = 需要最高级别的详尽分析

### 讨论后的更新
- 用户通过 Q&A 讨论后产生的新认识（如消融解读、方案比较），必须回流更新到对应的文档
- 更新时在原文下方加 callout 标注更新日期和链接，而非覆盖原文

### 实施方案
- 任何改进建议必须附带**分阶段实施路线**（Phase 0/1/2），从最小改动开始
- 每个 Phase 说明：工作量估算、预期收益（引用消融数据）、验证指标
- 代码示例用 PyG/PyTorch 风格，给出关键 API 调用

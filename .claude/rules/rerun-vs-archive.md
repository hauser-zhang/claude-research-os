# 实验重跑 vs 保留旧版（L2 · 跨项目通用）

> **定位**：修 bug、调作图风格、改 pipeline 后，要清旧结果重跑还是保留旧版新建带版本后缀？判据是**重跑成本**，不是"以防万一"。
> **适用**：所有科研项目的实验产出目录（`results/` / `figures/` / `analysis/` 等）。

---

## 核心判据：运行时间

| 运行时间 | 策略 | 理由 |
|---------|------|------|
| **< 10 分钟** | 清空当前任务目录下的旧输出 → 原地重跑覆盖 | 代码变更由 git 留痕，结果能快速重现；保留旧版只会让 `figures/` 堆 `_v2`/`_v3`，用户分不清哪张是最新 |
| **> 10 分钟 或 需要 GPU 训练** | 新建带版本/日期后缀的子目录，保留旧结果 | 重跑成本高；旧结果也需要能对比 |

---

## 清理规则（铁律）

### 1. 只清当前任务目录

改某个 case study 的 bug → 只清该 case study 自己的 `figures/` 目录。

**禁止**：
- 清其他 case study 的输出
- 清其他模型的输出
- 清其他 session 的输出（跨日期的产出不是本次 scope）

### 2. 复用 pipeline 内建清理逻辑

优先用 pipeline 自带的清理函数（如 `_cleanup_figures_dir`）。
如无 → 手动列出要删的 png / svg / csv，**逐个删除并打日志**（不要 `rm -rf`）。

### 3. 禁止 `_v2` / `_v3` 文件后缀

- 版本控制是 **git 的事**，不是文件系统的事
- 反例：改 bug 后生成 `figures/` 和 `figures_v2/` 并列 → 几周后用户回来不知道哪个对
- 正例：清掉旧的，原地重跑，git commit 留痕

---

## 典型案例

### ✅ 正确：pipeline bug 修复（短重跑路径）

- 修了某个 evaluator 的 filter bug
- pipeline 几十秒跑完
- 清旧 10 个 png/svg，原地重跑，不留 `_v2`
- git commit 记录 "fix: <具体 bug 描述>"

### ❌ 错误：堆版本后缀

- 改 bug 后生成 `figures/` + `figures_v2/` 并列
- 或者在文件名加 `_20260420` 后缀
- 过几周用户回来：不知道哪个对，要对比两份才能判断

---

## 决策流程

```
修完 bug / 改完风格 → 估算重跑时间
  │
  ├── < 10 分钟 → 清当前目录 → 原地重跑 → git commit
  │                    (禁止清其他目录 / 禁止加 _v2 后缀)
  │
  └── > 10 分钟 / 需 GPU → 建 <原目录>_<日期>/ 或 <原目录>_v2/
                         → 保留旧目录，新目录跑
                         → 在父目录 README.md 记录两个版本差异
```

## 反面教训速查

| 症状 | 根因 | 修复 |
|------|------|------|
| `figures/` 下一堆 `_alt` / `_v2` / `_v3` | 违反判据 1（<10min 仍加后缀） | 清旧，原地重跑 |
| 不同 case study 图同时消失 | 违反规则 1（清了不该清的目录） | 按目录隔离清理 |
| 清完发现原数据找不回来 | `rm -rf` 没打日志 | 逐文件 `rm` 并 `echo` 日志 |
| 过几周用户回来认错版本 | 有多个版本后缀并存 | 只保留最新版 + git log 对比 |
